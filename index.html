测试结论

  1. 批量建立索引的过程中，单台机器无副本的情况下CPU是性能的瓶颈。(通过监控看出CPU高负载主要表现在大量的上下文切换，并不表现在磁盘io等待。负载高导致ES无法响应)

ES 1.1.2

测试一：5个线程同时查询，共聚合查询50000次，完成每次查询平均耗时：0.06秒/次


测试二：每次清除缓存，单线程，聚合查询100次，完成每次查询平均耗时：3.45秒/次


ES 2.1.1

测试一：5个线程同时查询，共聚合查询50000次，完成每次查询平均耗时：0.38秒/次


测试二：每次清除缓存，单线程，聚合查询100次，完成每次查询平均耗时：4.27秒/次


测试结论

  1. 在【测试一】批量聚合的过程中ES 1.x的版本比ES 2.x的版本快很多，这是因为ES 2.x对系统的稳定和整体的执行效率要求更加的高。对数据缓存有了更加严格限制。
    从图【ES 1.1.2-Cache evictions】与图【ES 2.1.1-Cache evictions】对比可以看出ES 1.1.2的缓存失效非常的快，而ES 2.1.1的缓存失效确要慢得多。
    官方说明：https://www.elastic.co/guide/en/elasticsearch/guide/current/filter-caching.html#_autocaching_behavior
  2. 在【测试二】中ES 1.x与ES 2.x相差无几，ES 1.x中直接清除缓存对它的作用并不是很明显。

利用Github+Hexo搭建博客

发表于 2015-08-28   |   分类于 技术 ， 日常   |  

之前一直都想抽时间在github上弄一个博客，今天终于抽了一天的时间用github+hexo完成了搭建。

主要依赖的工具有：

Git
Node.js

安装Git与Node.js(Mac)

brew install git
brew install node

安装Hexo

# 安装hexo-cli相关的包
npm install hexo-cli -g 

# 初始化hexo相关的目录，生成相关目录及配置文件
# blog为生成的根目录的名字
hexo init blog 

cd blog
# 这里需要执行npm install，因为hexo init仅仅生成了一些初始化的内容，很多相关的插件与包其实是没有下载下来的，执行npm install会根据当前目录下的package.json去下载

npm install
# 启动服务，方便在本地查看效果

hexo server

发布内容到Gighub

# 安装部署文件到github的插件
npm install hexo-deployer-git --save

# 生成文件的同时部署到github
hexo generate --deploy
